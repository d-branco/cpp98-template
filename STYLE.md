# Coding Standards and Guidelines

This document outlines the mandatory coding standards and guidelines for this repository, with a strong emphasis on adherence to the C++98 standard and the 42 School Norm (v4).

## 1. Mandatory Standards

- **C++ Standard:** C++98 (ISO/IEC 14882:1998).
- **Norm Compliance:** 42 School Norm (v4).
- **Class Form:** Orthodox Canonical Form (OCF) is mandatory for all classes.
- **Memory Management:** No memory leaks are tolerated. All allocations must be paired with explicit deallocations (`new`/`delete`).

## 2. C++ Standard Enforcement

All code must be strictly compliant with the C++98 standard. This includes:
- Avoiding C++11/14/17/20 features (e.g., `auto`, range-based for loops, smart pointers, `std::string_view`, `nullptr`, etc.).
- Use `NULL` for null pointer assignments, as `nullptr` is forbidden.
- Using C-style casts or `static_cast`/`reinterpret_cast`/`const_cast`/`dynamic_cast` as appropriate (note: the build system strictly warns on C-style casts via `-Wold-style-cast`).
- Ensuring proper use of `<iostream>` with the `std::` namespace prefix.

### Forbidden Keywords
The C++ keywords `friend` and `namespace` are forbidden.

## 3. 42 School Norm (v4) Compliance

All code must comply with the 42 School Norm (v4). Key aspects include:
- **Header Guards:** Use standard `#ifndef`/`#define`/`#endif` guards in `FILENAME_HPP` format. All `#endif` directives must include a comment indicating what they are closing (e.g., `#endif // FILENAME_HPP`).
- **Function Length:** Functions should be concise and generally not exceed 25 lines.
- **Line Length:** Lines should not exceed 80 columns.
- **Magic Numbers:** Avoid magic numbers where possible by defining named constants. *(Note: automated linters are currently configured to ignore magic numbers, but the rule applies conceptually).*
- **Banners:** Do not manually edit the banners at the top of the `.*pp` files. The `Makefile` manages these automatically. To refresh headers manually, run `make headers`.

## 4. Naming Conventions

- **Classes / Enums / Structs:** `UpperCamelCase` (e.g., `UtilTime`).
- **Functions:** `snake_case` (e.g., `get_seconds`).
- **Variables:** `snake_case` with a minimum of 3 characters (e.g., `idx`, `count`).
- **Private Members:** `snake_case_` (e.g., `time_`).
- **Constants / Macros / Enum Values:** `SCREAMING_SNAKE_CASE` (e.g., `SECONDS_IN_HOUR`).
- **Filenames:** `UpperCamelCase` for files hosting a primary class; `snake_case` for generic utilities.

## 5. Orthodox Canonical Form (OCF)

The Orthodox Canonical Form is **mandatory** for all classes.

A class is in Orthodox Canonical Form if it explicitly defines the following four member functions:

1. **Default Constructor:** `ClassName();`
2. **Copy Constructor:** `ClassName(const ClassName& other);`
3. **Copy Assignment Operator:** `ClassName& operator=(const ClassName& other);`
4. **Destructor:** `~ClassName();`

These must be explicitly defined, even if the compiler-generated versions are sufficient.

## 6. Formatting & Tooling

- **Braces:** Allman style (always on a new line).
- **Indentation:** Tabs (width 4).
- **Line Length:** Maximum 80 columns.
- **Alignment:** Align consecutive declarations and assignments for readability.
- **Warning:** Do **NOT** manually edit `.clang-format`, `.clang-tidy`, or `.clangd`. These configuration files are dynamically generated by the `Makefile`. To modify formatting rules, update the respective targets in the `Makefile`.

## 7. Project Structure

- **Source Files:** `.cpp` extension.
- **Header Files:** `.hpp` extension.
- **Directory Layout:**
  - `srcs/`: Contains all implementation (`.cpp`) files.
  - `include/`: Contains all header (`.hpp`/`.h`) files.
  - `Makefile`: Standard build system.

## 8. Build Process & Compiler Flags

- **Build Tool:** `make`
- **Compiler:** `c++` or `clang++`
- **Mandatory Flags:** `-std=c++98 -Wall -Wextra -Werror`
- **Strict Linting Flags:** The Makefile additionally enforces `-Wshadow -Wnon-virtual-dtor -Wpedantic -Wconversion -Wsign-conversion -Wold-style-cast`. Code must compile without triggering any of these warnings.

## 9. Testing & Verification

- **Unit Testing:** Use `doctest` for unit testing. Tests must be guarded by `#ifdef TESTING`.
- **Memory Testing:** Memory leaks are strictly prohibited. The test suite automatically runs under `valgrind`.
- **Test Commands:**
  - `make test`: Runs compilation, unit tests, and valgrind memory checks.
  - `make valgrind`: Runs the standard executable wrapped in valgrind with full leak-check analysis.

## 10. Development Workflow

- **Format code:** `make format`
- **Check style:** `make style`
- **Apply fixes:** `make fix-style`
- **Generate Headers:** `make headers`

## 11. Include Directives & Dependencies

- **Libraries Definition:** Standard libraries and external dependencies must be included within the corresponding header file (`.hpp`), and **never** directly inside the `.cpp` file.
- **Source Includes:** Each `.cpp` file must include **only** its own matching header file.
- **Main Include:** The `main.cpp` file must solely include the central `header.hpp` header.
